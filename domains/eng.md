## AMM Rules (non-negotiable)

**The single most dangerous class of bug in Kasiro is inline pool math.**

The 2026-05-03 Oyebanji market incident was caused by copy-paste of AMM math that diverged from the canonical helper. This took down an active market.

### Rule: All AMM writes must use helpers from `server/amm.ts`

```typescript
// ALWAYS ŌĆö call the helper
import { poolsAfterBuy, poolsAfterSell } from './amm';
const { yesPool, noPool } = poolsAfterBuy(currentYesPool, currentNoPool, amount, 'YES');

// NEVER ŌĆö inline the constant-product math
const k = yesPool * noPool;
const newNoPool = k / (yesPool + amount); // FORBIDDEN
```

**Why:** The helper is the source of truth. Inline math diverges silently. Divergence = incorrect prices = unhappy traders = voided markets.

### AMM constants
- `feeBps: 200` ŌĆö 2% on every trade
- Constant-product invariant: `yes_pool * no_pool = k`
- Opening price set by operator in 0.0ŌĆō1.0 range (this is `openPrice`)
- Seeding done by `operatorSeedMarket` ŌĆö DO NOT bypass with SQL-direct activation

### AMM activation rule
SQL-direct activation (`UPDATE markets SET status='open'`) bypasses `operatorSeedMarket`. For AMM markets this leaves pools notional with no operator capital. Always use the admin dialog or call `operatorSeedMarket` programmatically.

### Pool re-pin after seeding
After `operatorSeedMarket`, 3 of 4 call sites re-pin pools. The unmasked vector was draftŌåÆopen PATCH. Re-pin is now added. Never remove the re-pin step.

## Parimutuel Rules

- `feeBps: 800` ŌĆö 8% rake (low vs 15ŌĆō25% industry norm)
- Seed weights per outcome must sum to exactly 100
- Pool closes at `closeAt`; no new trades after lock
- Participants share the final pool proportionally (minus rake)
- Seeding skips `operatorSeedMarket` entirely ŌĆö DO NOT add AMM seeding to parimutuel activation path

### Branching rule
Market mechanics type must always gate the activation path:

```typescript
if (market.mechanicsType === 'amm') {
  // seed pools via operatorSeedMarket
} else if (market.mechanicsType === 'parimutuel') {
  // no seeding ŌĆö just open
}
```

**Never collapse this branch.** The 2026-05-03 bug was partly caused by a collapsed activation path.

## Replit Publish DDL-Generator Bug (found 2026-07-09)

Replit's Publish flow diffs the **live dev database against the live prod database** (not `schema.ts` against a database) and auto-generates DDL to reconcile. Its generator for `ALTER TABLE ... ADD CONSTRAINT ... CHECK (...) NOT VALID` statements is broken: it appends a stray extra `)` after `NOT VALID` whenever the constraint expression contains ANY parenthesis character ŌĆö including ones inside a string literal (e.g. a regex like `'^(YES|NO|...)$'`), not just structural SQL parens. Cost 6 failed publish attempts to isolate on 2026-07-09.

**Workaround that works:** write CHECK constraints as paren-free expressions where possible ŌĆö e.g. use top-level `|` regex alternation (`'^YES$|^NO$|^[a-z][a-z0-9_]{0,63}$'`) instead of `= ANY (ARRAY[...])` or grouped alternation `(A|B|C)`. Verify the rewrite is semantically identical against real production data before applying (test all existing rows still match).

**If a constraint keeps breaking Publish:** decouple by reverting the **dev database state**, not just `schema.ts` ŌĆö Publish diffs live DB-to-DB, so reverting `schema.ts` alone doesn't remove it from the diff.

**No direct SQL path to production exists on this project** ŌĆö prod access is read-only outside of Publish's auto-diff. There is no manual escape hatch; a broken constraint expression can fully block deployment until rewritten paren-free.

**Known outstanding issue:** `parimutuel_buckets.outcome_format` constraint regex (`^[a-z][a-z0-9_]{0,63}$`) rejects 14 live digit-leading range-bucket values (e.g. `10_to_15`, `220k_to_250k`) generated by the range-market feature. Constraint predates that feature. Logged in `docs/KNOWN_ISSUES.md` ŌĆö do not run `VALIDATE CONSTRAINT` on it until the regex is widened in a separate reviewed change.

**Report this to Replit as a platform bug** ŌĆö it will hit the next person who adds any regex-based or array-based CHECK constraint on any Replit project, not just this one.

## Database Guardrails

### Raw SQL snake/camel drift
Three incidents in 2026-05-30 from this exact bug:
- Raw SQL uses `snake_case` column names
- Drizzle schema uses `camelCase`
- When you do `select *` or omit columns from `explicit-select()`, the mapping silently drops or misnames fields

**Rule:** Always use Drizzle's `explicit-select()` ŌĆö never `select *` in raw SQL or Drizzle queries. List every column you need. Undefined fields silently break the client.

### Auto-lock must demote promo_status
When a market transitions `open ŌåÆ locked`, the auto-lock job must atomically clear `feature` and `featured_rank`.

**Why:** Locked markets with feature badges leak into the ŌŁÉ Featured rail. This happened with the 2026-05-30 Rabat incident ŌĆö a locked market appeared featured for hours.

```typescript
// Required in auto-lock transition:
await db.update(markets)
  .set({ status: 'locked', feature: null, featured_rank: null })
  .where(eq(markets.id, marketId));
```

### operatorSeedMarket self-healing
3 of 4 call sites re-pin pools after seeding, masking potential pool state bugs. DraftŌåÆopen PATCH was the unmasked vector. Keep the re-pin. Do not optimise it away.

## Market Gates (v0.1, shipped 2026-05-22)

Two activation gates block poorly formed markets from going live:

| Gate | What it checks | Blocks on |
|---|---|---|
| `clarity_lint` | Market question is well-formed, unambiguous, has clear resolution | Vague questions, missing resolution criteria |
| `source_check` | At least one public URL provided that will definitively resolve the claim | Missing or unreachable source URL |
| `risk_compliance` | Banned market types, manipulation risk, settlement stability | Any banned category match |

**Pattern:** `risk_compliance` uses a saga/compensating-action pattern ŌĆö it rolls back if any risk check fails. Do not break the saga flow.
